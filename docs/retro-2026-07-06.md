## The 5 mechanisms we proved (were "has code, never fired")

The gap that mattered: every prior run was *too clean* to exercise the hard paths.
We built scenarios that deterministically force each one, then measured:

- **Rework loop** — goal said "in-memory only" → acceptance persistence check FAILed →
  worker read the failLines and switched `store.js` to `fs.writeFileSync` write-through →
  attempt 2 PASS. `reworks_triggered=1`. It doesn't retry blindly; it fixes the named failure.
- **Self-repair** — `maxTurns=1` starved the workers → critic rejected the stubs → retry.
  `self-repairs=4`.
- **Claim gate (clobber, build)** — two workers both told to write `shared.js` →
  first claims, second blocked. `prevented-clobbers=1, write-collisions=0, files-written=1`.
- **Path confinement** — a task told to write `../ESCAPED.txt` → the file never landed
  outside the workspace.
- **Graph memory** — `AURALIS_BUILD_GRAPH=1` → `graph.build=4` edge-batches + `graph.expand=4`
  used in recall.

## What worked (keep doing)

- **Validate the instrument first.** Before every build we ran the new acceptance spec
  against a known-good reference (5/5 PASS). So a later FAIL is the build's fault, not the
  spec's. This is what makes "measured, not asserted" trustworthy.
- **Gap matrix over vibes.** Listing every mechanism × proven?/not surfaced the 5 unexercised
  ones precisely, instead of guessing what to test.
- **Reuse seams, don't build.** Progress streaming was two edits to existing seams (the SDK
  `tool_use` loop and the timeline `emit`), not a new subsystem.
- **Relay progress, don't go dark.** Peeking the live stream at milestones and reporting
  "where it's at now" — now the default working style for long runs.

## What's still open (candidates, not committed)

- **Planner over-serializes on false dependencies.** In the restapi build, `router` took
  `store` injected (it never `require`s it) yet the planner made it depend on `store`, forcing
  sequential levels. Distinguish "needs to *know* an interface" (satisfiable via the brain,
  can run in parallel) from "needs to *require* the file" (must serialize). ~one worker's
  wall-time per build.
- **Acceptance specs are hardcoded per target** (`rps/todo/restapi/calc`). Deliberate: an
  objective PASS/FAIL needs a human-defined contract, and letting the same model write both
  the code and the spec would not be objective. Only revisit when the manual cost actually bites.
- Deferred in memory: per-task review gate + stories layer, cross-process claim registry
  (TTL/lease), oracle-lite auth, async job model for the MCP build tool, model routing.

## One-line takeaway

auralis now does analyze→build→verify→rework end-to-end, every mechanism is proven on a real
run, and it narrates itself while it works — the platform is real, not a demo.
